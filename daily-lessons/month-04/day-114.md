# Day 114: Transaction Isolation Levels

## Topic
Understanding the four transaction isolation levels and the concurrency problems they prevent: dirty reads, non-repeatable reads, and phantom reads.

## English Explanation
When multiple transactions run at the same time, they can interfere with each other. Isolation levels control how much transactions can see each other's uncommitted or changing data. There are four levels from least strict to most strict: READ UNCOMMITTED (can see uncommitted changes), READ COMMITTED (only sees committed data), REPEATABLE READ (same query gives same results within a transaction), and SERIALIZABLE (transactions behave as if run one after another). PostgreSQL defaults to READ COMMITTED. MySQL InnoDB defaults to REPEATABLE READ.

## SQL Example
```sql
-- Setting isolation level in PostgreSQL
BEGIN;
SET TRANSACTION ISOLATION LEVEL READ COMMITTED;
  SELECT balance FROM accounts WHERE account_id = 1;
  -- Another transaction might change this balance and commit
  -- If we SELECT again, we might see a different value
COMMIT;

-- REPEATABLE READ: same query, same result
BEGIN;
SET TRANSACTION ISOLATION LEVEL REPEATABLE READ;
  SELECT balance FROM accounts WHERE account_id = 1; -- Returns 5000
  -- Even if another transaction changes and commits, we still see 5000
  SELECT balance FROM accounts WHERE account_id = 1; -- Still 5000
COMMIT;

-- SERIALIZABLE: strictest level
BEGIN;
SET TRANSACTION ISOLATION LEVEL SERIALIZABLE;
  SELECT SUM(balance) FROM accounts;
  -- No other transaction can modify accounts until we finish
COMMIT;
```

## Technical Vocabulary

| Term | Definition |
|------|-----------|
| Dirty Read | Reading uncommitted data from another transaction |
| Non-repeatable Read | Same query returns different results within one transaction |
| Phantom Read | New rows appear in repeated query due to other transactions |
| READ UNCOMMITTED | Lowest isolation; allows dirty reads |
| READ COMMITTED | Only sees committed data (PostgreSQL default) |
| REPEATABLE READ | Consistent reads within transaction (MySQL default) |
| SERIALIZABLE | Highest isolation; transactions appear sequential |
| Concurrency | Multiple operations happening at the same time |

## Speaking Practice
Explain to a client: "We use isolation levels to balance speed and safety. For your financial reports, we use REPEATABLE READ so numbers stay consistent during report generation. For your real-time dashboard, we use READ COMMITTED which is faster but shows the latest data. This ensures reports are accurate while the dashboard stays responsive."

| Problem | READ UNCOMMITTED | READ COMMITTED | REPEATABLE READ | SERIALIZABLE |
|---------|:---:|:---:|:---:|:---:|
| Dirty Read | Possible | Prevented | Prevented | Prevented |
| Non-repeatable Read | Possible | Possible | Prevented | Prevented |
| Phantom Read | Possible | Possible | Possible* | Prevented |

*PostgreSQL prevents phantom reads at REPEATABLE READ level.
