# Day 101: Multiple CTEs and CTE vs Subquery Comparison

## Topic
Chaining multiple CTEs in one WITH block and comparing CTEs with nested subqueries for readability and maintainability.

## English Explanation
A Common Table Expression (CTE) uses the WITH keyword to create a temporary named result set. You can chain multiple CTEs in one WITH block by separating them with commas. Each later CTE can reference any earlier CTE defined above it. This makes complex queries much more readable than deeply nested subqueries. When you compare a nested subquery approach to a CTE approach for the same logic, CTEs win in clarity, debugging ease, and maintainability.

## SQL Example

### Multiple CTEs in One WITH Block
```sql
WITH active_customers AS (
    SELECT customer_id, first_name, last_name
    FROM customers
    WHERE status = 'active'
),
customer_orders AS (
    SELECT ac.customer_id, ac.first_name, COUNT(o.order_id) AS order_count
    FROM active_customers ac
    JOIN orders o ON ac.customer_id = o.customer_id
    GROUP BY ac.customer_id, ac.first_name
),
high_value_customers AS (
    SELECT customer_id, first_name, order_count
    FROM customer_orders
    WHERE order_count > 10
)
SELECT * FROM high_value_customers
ORDER BY order_count DESC;
```

### Same Logic as Nested Subquery (Hard to Read)
```sql
SELECT customer_id, first_name, order_count
FROM (
    SELECT ac.customer_id, ac.first_name, COUNT(o.order_id) AS order_count
    FROM (
        SELECT customer_id, first_name, last_name
        FROM customers
        WHERE status = 'active'
    ) ac
    JOIN orders o ON ac.customer_id = o.customer_id
    GROUP BY ac.customer_id, ac.first_name
) co
WHERE order_count > 10
ORDER BY order_count DESC;
```

## Technical Vocabulary

| Term | Definition |
|------|-----------|
| CTE | Common Table Expression - a named temporary result set |
| WITH clause | SQL keyword that introduces one or more CTEs |
| Chained CTE | Multiple CTEs separated by commas in one WITH block |
| Subquery | A query nested inside another query |
| Readability | How easy code is to understand for humans |
| Maintainability | How easy code is to modify later |
| Named result set | A temporary table-like structure with a name |
| Reference | When a later CTE uses an earlier CTE by name |

## Speaking Practice
"We use multiple CTEs instead of nested subqueries because each step has a clear name. This makes it easier for the team to debug and modify the query later. Think of CTEs as building blocks - each one does one thing clearly."

## Reading Practice
Read the PostgreSQL documentation on WITH queries. Notice how the docs show that CTEs are evaluated once and referenced by name, unlike subqueries which may be re-evaluated.
