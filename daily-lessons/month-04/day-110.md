# Day 110: Query Optimization Techniques

## Topic
Query Optimization Techniques

## English Explanation
Query optimization is the practice of writing SQL queries that execute as efficiently as possible. Key techniques include: **Avoid SELECT *** — always select only the columns you need, which reduces data transfer and can enable index-only scans. **Filter early** — apply WHERE conditions before JOINs when possible, reducing the number of rows that need to be joined. **Avoid functions on indexed columns** — writing `WHERE YEAR(date_col) = 2024` prevents index usage; instead use `WHERE date_col >= '2024-01-01' AND date_col < '2025-01-01'`. **Use LIMIT** for large result sets — if you only need the first 10 results, tell the database so it can stop early. **Use EXISTS instead of IN** for subqueries with large result sets. **Avoid unnecessary DISTINCT** — if your query logic is correct, you should not need DISTINCT to remove duplicates.

## SQL Example

```sql
-- BAD: SELECT * fetches all columns
SELECT * FROM orders WHERE customer_id = 5;

-- GOOD: Select only needed columns
SELECT order_id, order_date, total_amount FROM orders WHERE customer_id = 5;

-- BAD: Function on indexed column prevents index use
SELECT * FROM orders WHERE UPPER(status) = 'PENDING';

-- GOOD: Match the stored value directly
SELECT order_id, status FROM orders WHERE status = 'pending';

-- BAD: No LIMIT on large result set
SELECT order_id, total_amount FROM orders WHERE total_amount > 10 ORDER BY total_amount DESC;

-- GOOD: Use LIMIT to restrict results
SELECT order_id, total_amount FROM orders WHERE total_amount > 10
ORDER BY total_amount DESC LIMIT 20;

-- BAD: IN with large subquery
SELECT * FROM customers WHERE customer_id IN (SELECT customer_id FROM orders);

-- GOOD: EXISTS is often faster for large datasets
SELECT * FROM customers c
WHERE EXISTS (SELECT 1 FROM orders o WHERE o.customer_id = c.customer_id);
```


## Hands-on Practice
Rewrite these 5 queries for better performance:

1. Replace `SELECT * FROM orders JOIN customers ON ...` → select only order_id, order_date, name
2. Replace `WHERE EXTRACT(MONTH FROM order_date) = 3` → use date range
3. Add LIMIT to `SELECT order_id, total_amount FROM orders ORDER BY total_amount DESC`
4. Replace `WHERE customer_id IN (SELECT ...)` → use EXISTS
5. Filter orders by status='pending' before joining to customers

Run EXPLAIN on both original and optimized versions to compare costs.

## Technical Vocabulary

| Term | Definition |
|------|-----------|
| Query optimization | Writing queries to execute as efficiently as possible |
| Index-only scan | When all needed data comes from the index itself |
| Covering index | An index that contains all columns needed by a query |
| SARGable | A condition that allows the optimizer to use an index |
| EXISTS | Checks if a subquery returns any rows (often faster than IN) |
| LIMIT | Restricts the number of rows returned |
| Filter early | Applying conditions before joins to reduce row count |
| Execution plan | The strategy the database uses to execute a query |

## Reading Practice
Review these optimization principles:
- The database optimizer makes choices based on statistics
- ANALYZE command updates table statistics for better planning
- Sometimes the optimizer chooses Seq Scan because the table is small
- Premature optimization: only optimize queries that are actually slow
- Monitor slow query log to find queries that need attention
