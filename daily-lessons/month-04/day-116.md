# Day 116: Introduction to Stored Procedures

## Topic
Introduction to Stored Procedures — reusable blocks of SQL logic stored in the database.

## English Explanation
A stored procedure is a saved set of SQL statements that you can call by name. In PostgreSQL, we use `CREATE FUNCTION` with `LANGUAGE plpgsql` to create stored procedures. They accept parameters (IN for input, OUT for output, INOUT for both), encapsulate business logic, reduce network traffic by running on the server, and make code reusable across applications.

## SQL Example

```sql
-- Basic stored procedure in PostgreSQL
CREATE OR REPLACE FUNCTION add_new_order(
    p_customer_id INT,
    p_product_id INT,
    p_quantity INT
)
RETURNS INT AS $$
DECLARE
    new_order_id INT;
BEGIN
    INSERT INTO orders (customer_id, product_id, quantity, order_date)
    VALUES (p_customer_id, p_product_id, p_quantity, CURRENT_DATE)
    RETURNING order_id INTO new_order_id;

    RETURN new_order_id;
END;
$$ LANGUAGE plpgsql;

-- Calling the procedure
SELECT add_new_order(101, 55, 3);
```

## Technical Vocabulary

| Term | Definition |
|------|-----------|
| Stored Procedure | A saved block of SQL logic stored in the database |
| CREATE FUNCTION | PostgreSQL command to create a stored procedure |
| LANGUAGE plpgsql | Specifies PostgreSQL procedural language |
| IN parameter | Input-only parameter passed to the procedure |
| OUT parameter | Output-only parameter returned from the procedure |
| INOUT parameter | Parameter that serves as both input and output |
| RETURNS | Specifies the return data type |
| DECLARE | Section where local variables are defined |
| BEGIN/END | Marks the start and end of the procedure body |
| Encapsulation | Hiding complex logic behind a simple interface |

## Speaking Practice
"We use stored procedures to encapsulate reusable business logic inside the database. This reduces network traffic because the application sends one call instead of multiple queries. Parameters let us pass data in and get results out."
