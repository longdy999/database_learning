# Day 115: Error Handling - SAVEPOINT and Partial Rollback

## Topic
Using SAVEPOINT to create checkpoints within a transaction, allowing partial rollback without undoing the entire transaction.

## English Explanation
Sometimes a transaction has multiple steps, and you want to undo only part of it if something goes wrong. SAVEPOINT creates a named checkpoint inside a transaction. If a later step fails, you can ROLLBACK TO SAVEPOINT to undo only the work after that savepoint, then continue the transaction. This is essential for complex operations like order processing where some items might be out of stock but you still want to process the available ones.

## SQL Example
```sql
-- Basic SAVEPOINT syntax
BEGIN;
  INSERT INTO orders (customer_id, total) VALUES (1, 0);
  
  SAVEPOINT before_items;
  
  INSERT INTO order_items (order_id, product_id, qty, price)
  VALUES (LASTVAL(), 101, 2, 25.00);
  UPDATE inventory SET stock = stock - 2 WHERE product_id = 101;
  
  SAVEPOINT after_first_item;
  
  -- This might fail if product 102 is out of stock
  INSERT INTO order_items (order_id, product_id, qty, price)
  VALUES (LASTVAL(), 102, 1, 50.00);
  UPDATE inventory SET stock = stock - 1 WHERE product_id = 102;
  
  -- If product 102 failed, rollback only that part:
  -- ROLLBACK TO SAVEPOINT after_first_item;
  
  -- Update order total and commit what worked
  UPDATE orders SET total = 50.00 WHERE order_id = LASTVAL();
COMMIT;
```


## Hands-on Practice
**Task: Write a transaction with savepoints for an order process**

```sql
-- Setup tables
CREATE TABLE inventory (
    product_id INT PRIMARY KEY,
    product_name VARCHAR(100),
    stock INT NOT NULL CHECK (stock >= 0)
);

INSERT INTO inventory VALUES
(101, 'Keyboard', 10),
(102, 'Mouse', 0),      -- Out of stock!
(103, 'Monitor', 5);

-- Write a transaction that:
-- 1. Creates an order
-- 2. Tries to add Keyboard (should succeed)
-- 3. SAVEPOINT
-- 4. Tries to add Mouse (should fail - out of stock)
-- 5. ROLLBACK TO SAVEPOINT (undo only Mouse attempt)
-- 6. Tries to add Monitor (should succeed)
-- 7. COMMIT the order with Keyboard + Monitor

BEGIN;
  SAVEPOINT start_order;
  -- Add keyboard
  UPDATE inventory SET stock = stock - 1 WHERE product_id = 101;
  
  SAVEPOINT after_keyboard;
  -- Try mouse (will fail due to CHECK constraint)
  UPDATE inventory SET stock = stock - 1 WHERE product_id = 102;
  -- If error: ROLLBACK TO SAVEPOINT after_keyboard;
  
  SAVEPOINT after_mouse_attempt;
  -- Add monitor
  UPDATE inventory SET stock = stock - 1 WHERE product_id = 103;
COMMIT;
```

## Technical Vocabulary

| Term | Definition |
|------|-----------|
| SAVEPOINT | Named checkpoint within a transaction |
| ROLLBACK TO SAVEPOINT | Undo work back to a specific savepoint |
| RELEASE SAVEPOINT | Remove a savepoint (optional cleanup) |
| Partial Rollback | Undoing only part of a transaction |
| Nested Checkpoint | Savepoints within savepoints |
| Error Recovery | Handling failures without losing all work |
| Graceful Degradation | Processing what you can, skipping what fails |
| Idempotent | Operation that produces same result if repeated |

## Reading Practice
Read this transaction and determine the final state:
```sql
BEGIN;
  UPDATE accounts SET balance = balance - 100 WHERE account_id = 1;
  SAVEPOINT sp1;
  UPDATE accounts SET balance = balance - 200 WHERE account_id = 1;
  SAVEPOINT sp2;
  UPDATE accounts SET balance = balance - 300 WHERE account_id = 1;
  ROLLBACK TO SAVEPOINT sp1;
  UPDATE accounts SET balance = balance - 50 WHERE account_id = 1;
COMMIT;
```
If starting balance was 1000, what is the final balance?
