# Day 108: EXPLAIN - Reading Query Execution Plans

## Topic
EXPLAIN - Reading Query Execution Plans

## English Explanation
EXPLAIN is a command that shows you how the database plans to execute a query without actually running it. It reveals whether the database will use an index, perform a full table scan, or use other strategies. This is the most important tool for understanding query performance. Key terms to understand: **Seq Scan** means the database reads every row in the table (slow for large tables). **Index Scan** means it uses an index to find rows quickly. **cost** is an estimate of how expensive the operation is. **rows** is the estimated number of rows the operation will return. **width** is the average size of each row in bytes. By reading EXPLAIN output, you can identify slow queries and decide where to add indexes.

## SQL Example

```sql
-- Basic EXPLAIN
EXPLAIN SELECT * FROM orders WHERE customer_id = 5;

-- Sample output:
-- Index Scan using idx_orders_customer on orders  (cost=0.15..8.17 rows=1 width=52)
--   Index Cond: (customer_id = 5)

-- EXPLAIN without index (Seq Scan example)
EXPLAIN SELECT * FROM orders WHERE shipping_city = 'Phnom Penh';

-- Sample output:
-- Seq Scan on orders  (cost=0.00..25.00 rows=5 width=52)
--   Filter: ((shipping_city)::text = 'Phnom Penh'::text)

-- EXPLAIN ANALYZE - actually runs the query and shows real timing
EXPLAIN ANALYZE SELECT * FROM orders WHERE customer_id = 5;

-- Sample output:
-- Index Scan using idx_orders_customer on orders
--   (cost=0.15..8.17 rows=1 width=52)
--   (actual time=0.025..0.027 rows=1 loops=1)
-- Planning Time: 0.085 ms
-- Execution Time: 0.045 ms

-- EXPLAIN with JOIN
EXPLAIN SELECT o.order_id, c.name
FROM orders o
JOIN customers c ON o.customer_id = c.customer_id
WHERE o.order_date > '2024-01-01';
```


## Hands-on Practice
1. Run `EXPLAIN SELECT * FROM orders WHERE customer_id = 10;` — identify if Index Scan or Seq Scan.
2. Run `EXPLAIN SELECT * FROM orders WHERE total_amount > 100;` — check if an index is used.
3. Run `EXPLAIN ANALYZE SELECT * FROM orders WHERE status = 'pending';` — note actual time.
4. Run `EXPLAIN` on a JOIN query between orders and customers.
5. Drop an index, run the same EXPLAIN, and compare the change in plan.
6. Write down: for each query, is it using Index Scan or Seq Scan?

## Technical Vocabulary

| Term | Definition |
|------|-----------|
| EXPLAIN | Command that shows query execution plan without running it |
| EXPLAIN ANALYZE | Runs the query and shows actual execution time |
| Seq Scan | Sequential scan; reads every row in the table |
| Index Scan | Uses an index to find matching rows quickly |
| cost | Estimated computational expense (startup..total) |
| rows | Estimated number of rows returned |
| width | Average row size in bytes |
| Filter | A condition applied after scanning rows |
| Planning Time | Time spent creating the execution plan |
| Execution Time | Actual time to run the query |

## Speaking Practice
"I ran EXPLAIN on the customer lookup query and it shows an Index Scan using idx_orders_customer. The estimated cost is very low at 0.15 to 8.17, and it expects to return just one row. This means our index is working correctly. However, the shipping_city query shows a Seq Scan, which means the database is reading every row. We should consider adding an index on shipping_city if this query runs frequently."

## Reading Practice
Read PostgreSQL EXPLAIN documentation:
- Understand the difference between EXPLAIN and EXPLAIN ANALYZE
- Learn to read cost as (startup_cost..total_cost)
- Recognize that Seq Scan on small tables is acceptable
- Identify when a Seq Scan on a large table indicates a missing index
