# Day 109: Identifying and Fixing Slow Queries

## Topic
Identifying and Fixing Slow Queries

## English Explanation
Slow queries are one of the most common database problems in production systems. The main causes include: **missing indexes** (the database must scan every row), **full table scans** on large tables, **poorly written JOINs** (joining too many tables or missing join conditions), **SELECT *** (fetching unnecessary columns), and **functions on indexed columns** (which prevent index usage). To identify slow queries, look for: high cost values in EXPLAIN output, Seq Scan on tables with thousands or millions of rows, and queries that return too many rows. Common fixes include: adding appropriate indexes, rewriting queries to be more specific, using LIMIT to reduce result sets, and avoiding unnecessary joins or subqueries.

## SQL Example

```sql
-- SLOW: Full table scan, no index on shipping_city
EXPLAIN ANALYZE
SELECT * FROM orders WHERE shipping_city = 'Phnom Penh';
-- Result: Seq Scan on orders (cost=0.00..1250.00 rows=500 width=52)

-- FIX: Add an index
CREATE INDEX idx_orders_city ON orders(shipping_city);

-- Now the same query uses Index Scan
EXPLAIN ANALYZE
SELECT * FROM orders WHERE shipping_city = 'Phnom Penh';
-- Result: Index Scan using idx_orders_city (cost=0.28..12.50 rows=500 width=52)

-- SLOW: Function on indexed column prevents index usage
SELECT * FROM orders WHERE LOWER(status) = 'pending';

-- FIX: Use expression index or match case directly
CREATE INDEX idx_orders_status_lower ON orders(LOWER(status));
-- OR simply: WHERE status = 'pending'

-- SLOW: SELECT * fetches all columns unnecessarily
SELECT * FROM orders WHERE customer_id = 5;

-- FIX: Select only needed columns
SELECT order_id, order_date, total_amount FROM orders WHERE customer_id = 5;
```

## Technical Vocabulary

| Term | Definition |
|------|-----------|
| Slow query | A query that takes too long to execute |
| Full table scan | Reading every row in a table (Seq Scan) |
| Missing index | No index exists for the columns being filtered |
| Query optimization | Rewriting queries or adding indexes to improve speed |
| Nested Loop | A join method that can be slow for large tables |
| Cost estimate | The database planner's prediction of query expense |
| Index-friendly | A query written in a way that allows index usage |
| SARGable | Search ARGument ABLE; a condition that can use an index |

## Reading Practice
Review common slow query patterns:
- Missing WHERE clause (returns too many rows)
- OR conditions that prevent index usage
- LIKE '%text%' (leading wildcard cannot use B-tree index)
- Implicit type conversion in WHERE clause
- Correlated subqueries that run once per row
