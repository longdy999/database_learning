# Day 113: ACID Properties Explained

## Topic
Understanding the four guarantees every database transaction provides: Atomicity, Consistency, Isolation, and Durability (ACID).

## English Explanation
ACID is the set of four properties that guarantee database transactions are reliable. Atomicity means all-or-nothing: every statement in a transaction either completes or none do. Consistency means the database moves from one valid state to another valid state (constraints are never violated). Isolation means concurrent transactions do not interfere with each other. Durability means once a transaction is committed, the data survives even if the server crashes immediately after.

## SQL Example
```sql
-- ATOMICITY example: all or nothing
BEGIN;
  INSERT INTO orders (customer_id, total) VALUES (1, 500.00);
  INSERT INTO order_items (order_id, product_id, qty) VALUES (LASTVAL(), 10, 2);
  UPDATE inventory SET stock = stock - 2 WHERE product_id = 10;
  -- If ANY of these fail, ALL are rolled back
COMMIT;

-- CONSISTENCY example: constraints protect valid state
CREATE TABLE employees (
    emp_id SERIAL PRIMARY KEY,
    salary DECIMAL(10,2) CHECK (salary > 0),
    department_id INT REFERENCES departments(dept_id)
);

-- This will fail - maintains consistency
BEGIN;
  UPDATE employees SET salary = -100 WHERE emp_id = 1;
  -- ERROR: violates check constraint
ROLLBACK;

-- DURABILITY: after COMMIT, data is permanent
BEGIN;
  INSERT INTO audit_log (action, timestamp) VALUES ('payment_processed', NOW());
COMMIT;
-- Even if server crashes NOW, this record exists
```

## Technical Vocabulary

| Term | Definition |
|------|-----------|
| ACID | Atomicity, Consistency, Isolation, Durability |
| Atomicity | All operations complete or none do |
| Consistency | Database always in a valid state |
| Isolation | Concurrent transactions don't interfere |
| Durability | Committed data survives system failures |
| WAL (Write-Ahead Log) | Mechanism that ensures durability |
| Constraint | Rule that enforces consistency |
| Concurrent | Multiple transactions running at the same time |

## Speaking Practice
Explain to a client: "Our database follows ACID principles. This means your data is always safe. If a multi-step operation fails halfway, nothing is partially saved. The system ensures data is always valid, different users don't corrupt each other's work, and once saved, data cannot be lost even during power outages."

## Reading Practice
Read this scenario and identify the ACID violation:
"A user transfers money. The debit succeeds and is saved, but the credit fails. The system does not undo the debit. The sender loses money but the receiver gets nothing."

Which ACID property was violated? How would a proper transaction prevent this?
