# Day 107: Index Types and When to Use Each

## Topic
Index Types and When to Use Each

## English Explanation
Not all indexes are the same. Different index types are optimized for different kinds of queries. The most common type is the **B-tree index**, which is the default in PostgreSQL and MySQL. B-tree indexes work well for equality comparisons (`=`) and range queries (`<`, `>`, `BETWEEN`, `LIKE 'prefix%'`). **Hash indexes** are optimized only for equality comparisons and cannot help with range queries. **Partial indexes** cover only a subset of rows (using a WHERE clause in the index definition), saving space and improving performance for specific queries. **Expression indexes** index the result of a function or expression rather than a raw column value. Choosing the right index type can significantly improve query performance.

## SQL Example

```sql
-- B-tree index (default) - good for ranges and equality
CREATE INDEX idx_orders_date ON orders(order_date);

-- This query benefits from B-tree:
SELECT * FROM orders WHERE order_date BETWEEN '2024-01-01' AND '2024-03-31';

-- Hash index - equality only (PostgreSQL syntax)
CREATE INDEX idx_orders_status_hash ON orders USING hash(status);

-- This query benefits from hash index:
SELECT * FROM orders WHERE status = 'shipped';

-- Partial index - index only active orders
CREATE INDEX idx_orders_active ON orders(customer_id)
WHERE status = 'pending';

-- This query benefits from partial index:
SELECT * FROM orders WHERE status = 'pending' AND customer_id = 42;

-- Expression index - index on lowercase email
CREATE INDEX idx_customers_email_lower ON customers(LOWER(email));

-- This query benefits from expression index:
SELECT * FROM customers WHERE LOWER(email) = 'john@example.com';
```

## Technical Vocabulary

| Term | Definition |
|------|-----------|
| B-tree index | Default index type; supports equality and range queries |
| Hash index | Index type optimized only for equality comparisons |
| Partial index | An index that covers only rows matching a condition |
| Expression index | An index on a computed expression, not a raw column |
| USING hash | Syntax to specify hash index type in PostgreSQL |
| Range query | A query with conditions like BETWEEN, <, >, <=, >= |
| Equality query | A query that checks exact match with = |
| Index selectivity | How well an index narrows down results |

## Speaking Practice
"For this project, I recommend using a B-tree index on the order_date column because we frequently run date range queries. For the status column, a hash index could work since we only check for exact values like 'pending' or 'shipped'. I also created a partial index that only covers pending orders, which saves space and speeds up our most common lookup."

## Reading Practice
Read about index types in the PostgreSQL documentation:
- B-tree: default, most versatile, supports `<`, `<=`, `=`, `>=`, `>`, `BETWEEN`, `LIKE 'prefix%'`
- Hash: only `=` operator, smaller in some cases
- GIN and GiST: for full-text search and geometric data (advanced, overview only)
- Understand that MySQL primarily uses B-tree and does not support hash indexes the same way.
