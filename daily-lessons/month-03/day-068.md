# Day 68: Why Normalization Matters - Data Anomalies

## Topic
Understanding insert anomalies, update anomalies, and delete anomalies in poorly designed tables.

## English Explanation
When a table stores too much information in one place, problems called "anomalies" occur. An insert anomaly means you cannot add new data without adding unrelated data. An update anomaly means changing one fact requires updating many rows. A delete anomaly means removing one record accidentally destroys other important data. Normalization solves these by splitting tables properly.

## SQL Example

```sql
-- BADLY DESIGNED TABLE (denormalized - all data in one table)
CREATE TABLE student_courses_bad (
    student_id INT,
    student_name VARCHAR(100),
    student_email VARCHAR(100),
    course_id INT,
    course_name VARCHAR(100),
    instructor VARCHAR(100),
    instructor_office VARCHAR(50),
    grade CHAR(2)
);

INSERT INTO student_courses_bad VALUES
(1, 'Sokha', 'sokha@mail.com', 101, 'SQL Basics', 'Dr. Chan', 'Room 201', 'A'),
(1, 'Sokha', 'sokha@mail.com', 102, 'Web Dev', 'Dr. Lim', 'Room 305', 'B'),
(2, 'Dara', 'dara@mail.com', 101, 'SQL Basics', 'Dr. Chan', 'Room 201', 'B+'),
(3, 'Vanna', 'vanna@mail.com', 101, 'SQL Basics', 'Dr. Chan', 'Room 201', 'A-');


-- INSERT ANOMALY: Cannot add a new course without a student
-- We want to add course 103 'Data Science' by Dr. Mao
-- But we need a student_id and student_name too!

-- UPDATE ANOMALY: Dr. Chan moves to Room 401
-- Must update EVERY row where Dr. Chan appears
UPDATE student_courses_bad
SET instructor_office = 'Room 401'
WHERE instructor = 'Dr. Chan';
-- If we miss one row, data becomes inconsistent!

-- DELETE ANOMALY: Dara drops out, we delete his record
DELETE FROM student_courses_bad WHERE student_id = 2;
-- If Dara was the only student in a course, we lose that course info!

-- SOLUTION: Split into proper normalized tables
CREATE TABLE students (
    student_id INT PRIMARY KEY,
    student_name VARCHAR(100),
    student_email VARCHAR(100)
);
CREATE TABLE courses (
    course_id INT PRIMARY KEY,
    course_name VARCHAR(100),
    instructor VARCHAR(100),
    instructor_office VARCHAR(50)
);
CREATE TABLE enrollments (
    student_id INT,
    course_id INT,
    grade CHAR(2),
    PRIMARY KEY (student_id, course_id),
    FOREIGN KEY (student_id) REFERENCES students(student_id),
    FOREIGN KEY (course_id) REFERENCES courses(course_id)
);
```

## Technical Vocabulary

| Term | Definition |
|------|-----------|
| Data Anomaly | Problem caused by poor table design |
| Insert Anomaly | Cannot add data without unrelated information |
| Update Anomaly | Changing one fact requires multiple row updates |
| Delete Anomaly | Removing a record destroys unrelated data |
| Denormalized | Table with redundant, repeated data |
| Normalization | Process of splitting tables to reduce redundancy |
| Redundancy | Same data stored in multiple places unnecessarily |

## Speaking Practice
"This table has serious design problems. If I need to update the instructor's office, I have to change it in every row where that instructor appears. If I miss one, the data becomes inconsistent. By splitting this into three tables вҖ” students, courses, and enrollments вҖ” each fact is stored only once."

## Reading Practice
Look at any large spreadsheet or flat table and identify: Which columns repeat the same values? Which groups of columns always appear together? These patterns indicate the table needs normalization.
