# Day 73: When NOT to Normalize (Denormalization)

## Topic
Denormalization — Strategic Decisions to Break Normal Form Rules for Performance

## English Explanation
Normalization is essential for data integrity, but sometimes we intentionally break normal form rules for better performance. This is called **denormalization**.

**When to denormalize:**
- **Read-heavy systems** — When a system reads data 100x more than it writes (e.g., news websites, dashboards)
- **Reporting databases** — When complex JOINs slow down reports that need instant results
- **Caching tables** — Pre-computed summary tables for fast access
- **Data warehouses** — Star/snowflake schemas are intentionally denormalized

**The tradeoff:**
- Normalization → Better data integrity, less redundancy, slower reads (more JOINs)
- Denormalization → Faster reads, simpler queries, but risk of data inconsistency and more storage

**Key rule:** Always normalize first, then denormalize specific parts only when you have a proven performance problem.

## Khmer Explanation
Normalization សំខាន់សម្រាប់ data integrity ប៉ុន្តែពេលខ្លះយើងចេតនាបំពាន normal form rules ដើម្បីបង្កើន performance។ នេះហៅថា **denormalization**។

**ពេលណាគួរ denormalize:**
- **ប្រព័ន្ធអានច្រើន** — ពេលប្រព័ន្ធអានទិន្នន័យ 100 ដងច្រើនជាងសរសេរ
- **មូលដ្ឋានទិន្នន័យរបាយការណ៍** — ពេល JOINs ស្មុគស្មាញធ្វើឱ្យរបាយការណ៍យឺត
- **តារាង caching** — តារាងសង្ខេបគណនាជាមុន សម្រាប់ចូលប្រើលឿន
- **Data warehouses** — Star/snowflake schemas ត្រូវបាន denormalize ចេតនា

**ការតូទៅវិញតូទៅមក:**
- Normalization → Data integrity ល្អជាង ប៉ុន្តែអានយឺតជាង (JOINs ច្រើន)
- Denormalization → អានលឿនជាង ប៉ុន្តែហានិភ័យទិន្នន័យមិនស៊ីសង្វាក់គ្នា

**ច្បាប់សំខាន់:** Normalize ជាមុនសិន បន្ទាប់មក denormalize ផ្នែកជាក់លាក់តែពេលមានបញ្ហា performance ពិតប្រាកដ។

## SQL Example

### Normalized Version (slower for reports):
```sql
-- To get order total, we need JOIN across 3 tables:
SELECT o.order_id, c.customer_name, SUM(p.price * oi.quantity) AS total
FROM orders o
JOIN customers c ON o.customer_id = c.customer_id
JOIN order_items oi ON o.order_id = oi.order_id
JOIN products p ON oi.product_id = p.product_id
GROUP BY o.order_id, c.customer_name;
```


### Denormalized Version (faster for reports):
```sql
-- Add redundant columns to orders table for quick access:
CREATE TABLE orders_denormalized (
    order_id INT PRIMARY KEY,
    order_date DATE,
    customer_id INT,
    customer_name VARCHAR(100),   -- redundant, but fast to read
    order_total DECIMAL(10,2),    -- pre-calculated, avoids JOINs
    item_count INT                -- pre-calculated summary
);

-- Now the report query is simple and fast:
SELECT order_id, customer_name, order_total
FROM orders_denormalized
WHERE order_date >= '2024-01-01';
```

### Summary Table (common denormalization pattern):
```sql
CREATE TABLE monthly_sales_summary (
    month DATE,
    product_id INT,
    product_name VARCHAR(100),
    total_sold INT,
    total_revenue DECIMAL(10,2),
    PRIMARY KEY (month, product_id)
);
```

## Speaking Practice
"In most cases, we normalize our database to prevent data inconsistency. However, for this reporting dashboard, we chose to denormalize the sales summary table. The dashboard is read thousands of times per hour but updated only once per day. The tradeoff is worth it — we accept slight redundancy in exchange for much faster query performance. We update the summary table with a scheduled job every night."
