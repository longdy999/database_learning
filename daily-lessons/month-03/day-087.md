# Day 87: Design Review - Spotting Problems

## Topic
Identifying common database design problems and learning to review schemas critically.

## English Explanation
Common design problems include: missing primary keys, no foreign keys between related tables, unnecessary denormalization, unclear or inconsistent naming, missing constraints, redundant data storage, and no indexes on frequently queried columns. A design review checks each table for these issues systematically.

## SQL Example
```sql
-- BAD DESIGN: Review and spot all problems
CREATE TABLE emp (
    name VARCHAR(100),
    dept VARCHAR(100),
    mgr_name VARCHAR(100),
    mgr_email VARCHAR(100),
    salary DECIMAL,
    proj1 VARCHAR(100),
    proj2 VARCHAR(100),
    proj3 VARCHAR(100)
);

-- Problems found:
-- 1. No PRIMARY KEY
-- 2. No FOREIGN KEY relationships
-- 3. Manager info repeated (denormalized without reason)
-- 4. Fixed number of project columns (not scalable)
-- 5. No NOT NULL constraints
-- 6. Vague column names (name of what?)
-- 7. No data types specified precisely
```

## Technical Vocabulary

| Term | Definition |
|------|-----------|
| Design Review | Systematic check of database structure for problems |
| Missing PK | A table without a primary key identifier |
| Denormalization | Storing redundant data, sometimes intentionally |
| Repeating Groups | Multiple columns for the same type of data (proj1, proj2) |
| Data Redundancy | Same information stored in multiple places |
| Constraint Violation | Missing rules that should protect data integrity |
| Scalability Issue | Design that breaks when data grows |
| Code Smell | A pattern that suggests deeper design problems |

## Speaking Practice
"When I review this database design, I can see several problems. First, there is no primary key, which means we cannot uniquely identify records. Second, the repeating columns like proj1, proj2, proj3 indicate a normalization issue — this should be a separate table. Third, manager information is duplicated across rows, causing data redundancy."

## Reading Practice
Review this design checklist and apply it to any schema:
- Does every table have a primary key?
- Are relationships defined with foreign keys?
- Is there unnecessary data duplication?
- Are column names clear and consistent?
- Are appropriate constraints (NOT NULL, UNIQUE, CHECK) applied?
- Could repeating column groups be moved to a separate table?
