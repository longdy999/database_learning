## SQL Example
```sql
-- Review: Apply all patterns learned this week
-- 1. Lookup table pattern
CREATE TABLE order_statuses (
    status_id SERIAL PRIMARY KEY,
    status_name VARCHAR(30) UNIQUE NOT NULL
);

-- 2. Audit trail pattern
CREATE TABLE orders (
    order_id SERIAL PRIMARY KEY,
    customer_id INT NOT NULL REFERENCES customers(customer_id),
    status_id INT REFERENCES order_statuses(status_id),
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    created_by INT REFERENCES users(user_id)
);

-- 3. Soft delete pattern
ALTER TABLE orders ADD COLUMN is_deleted BOOLEAN DEFAULT FALSE;
ALTER TABLE orders ADD COLUMN deleted_at TIMESTAMP;
```

## Technical Vocabulary

| Term | Definition |
|------|-----------|
| Checkpoint | A review point to verify understanding |
| Self-assessment | Evaluating your own knowledge honestly |
| Design Pattern | A reusable solution to a common problem |
| Audit Trail | Record of who changed what and when |
| Soft Delete | Marking records as deleted without removing them |
| Lookup Table | A table storing predefined valid values |
| Naming Convention | Consistent rules for naming database objects |
| Schema Review | Critical examination of database structure |

## Reading Practice
Answer these 10 design review questions:
1. What is a lookup table and when should you use one?
2. Why are audit fields (created_at, updated_at) important?
3. What is the difference between hard delete and soft delete?
4. Name 3 rules for professional table naming.
5. What pattern should foreign key columns follow?
6. Why should you avoid repeating column groups (col1, col2, col3)?
7. What is the first thing to check in a design review?
8. How do you document a database for your team?
9. When is denormalization acceptable?
10. What makes a column name "self-documenting"?
