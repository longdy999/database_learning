# Day 77: ERD Practice - Multiple Scenarios

## Topic
Drawing ERDs for Multiple Real-World Scenarios

## English Explanation
Today we practice drawing ERDs for different business domains. Each domain has common patterns that repeat across many database designs. By practicing multiple scenarios, you learn to recognize these patterns quickly.

**Common patterns in database design:**
- User/Account pattern: users, profiles, roles
- Transaction pattern: orders, payments, invoices
- Content pattern: posts, comments, likes
- Scheduling pattern: appointments, events, bookings

## SQL Example

**Scenario 1: Hospital System**
```sql
CREATE TABLE doctors (
    doctor_id SERIAL PRIMARY KEY,
    full_name VARCHAR(100) NOT NULL,
    specialty VARCHAR(50),
    phone VARCHAR(20)
);

CREATE TABLE patients (
    patient_id SERIAL PRIMARY KEY,
    full_name VARCHAR(100) NOT NULL,
    date_of_birth DATE,
    blood_type VARCHAR(5)
);


CREATE TABLE appointments (
    appointment_id SERIAL PRIMARY KEY,
    doctor_id INT REFERENCES doctors(doctor_id),
    patient_id INT REFERENCES patients(patient_id),
    appointment_date TIMESTAMP NOT NULL,
    diagnosis TEXT
);
```

**Scenario 2: Online Store**
```sql
CREATE TABLE customers (
    customer_id SERIAL PRIMARY KEY,
    username VARCHAR(50) UNIQUE NOT NULL,
    email VARCHAR(100) UNIQUE NOT NULL
);

CREATE TABLE products (
    product_id SERIAL PRIMARY KEY,
    product_name VARCHAR(100) NOT NULL,
    price DECIMAL(10,2) NOT NULL,
    stock_qty INT DEFAULT 0
);

CREATE TABLE orders (
    order_id SERIAL PRIMARY KEY,
    customer_id INT REFERENCES customers(customer_id),
    order_date TIMESTAMP DEFAULT NOW(),
    total_amount DECIMAL(10,2)
);

CREATE TABLE order_items (
    item_id SERIAL PRIMARY KEY,
    order_id INT REFERENCES orders(order_id),
    product_id INT REFERENCES products(product_id),
    quantity INT NOT NULL,
    unit_price DECIMAL(10,2) NOT NULL
);
```

**Scenario 3: Social Media Platform**
```sql
CREATE TABLE users (
    user_id SERIAL PRIMARY KEY,
    username VARCHAR(50) UNIQUE NOT NULL,
    bio TEXT,
    created_at TIMESTAMP DEFAULT NOW()
);


CREATE TABLE posts (
    post_id SERIAL PRIMARY KEY,
    user_id INT REFERENCES users(user_id),
    content TEXT NOT NULL,
    posted_at TIMESTAMP DEFAULT NOW()
);

CREATE TABLE comments (
    comment_id SERIAL PRIMARY KEY,
    post_id INT REFERENCES posts(post_id),
    user_id INT REFERENCES users(user_id),
    comment_text TEXT NOT NULL,
    commented_at TIMESTAMP DEFAULT NOW()
);

CREATE TABLE likes (
    like_id SERIAL PRIMARY KEY,
    post_id INT REFERENCES posts(post_id),
    user_id INT REFERENCES users(user_id),
    liked_at TIMESTAMP DEFAULT NOW(),
    UNIQUE(post_id, user_id)
);
```

## Speaking Practice
"When designing a database for any business, I look for common patterns. For example, an online store always needs customers, products, orders, and order items. A social media platform needs users, posts, comments, and likes. Recognizing these patterns helps me design faster and more reliably."

## Reading Practice
Compare the three scenarios above. Notice:
- Each has a "main entity" (doctors, customers, users)
- Each has a "transaction/action" table (appointments, orders, posts)
- Each uses foreign keys to connect entities
- The online store uses a junction table (order_items) for many-to-many

## Review Task

- Write one more scenario of your own choice and create its ERD
- Verify all foreign keys are correct in your design
