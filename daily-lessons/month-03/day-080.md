# Day 80: DEFAULT and Cascading Constraints Together

## Topic
Combining DEFAULT, NOT NULL, CHECK, UNIQUE, and Foreign Key Constraints in One Table

## English Explanation
In real-world database design, tables use multiple constraint types together. A professional table design combines:
- **PRIMARY KEY** — identifies each row uniquely
- **NOT NULL** — requires a value
- **DEFAULT** — provides automatic value if none given
- **CHECK** — validates data against rules
- **UNIQUE** — prevents duplicates
- **FOREIGN KEY** — links to another table

The key is knowing when to use each and how they work together.

- **FOREIGN KEY** — ភ្ជាប់ទៅតារាងមួយទៀត

## SQL Example
```sql
-- First, create referenced tables
CREATE TABLE departments (
    department_id SERIAL PRIMARY KEY,
    dept_name VARCHAR(50) UNIQUE NOT NULL,
    location VARCHAR(100) DEFAULT 'Main Office'
);

CREATE TABLE job_titles (
    job_id SERIAL PRIMARY KEY,
    title VARCHAR(100) UNIQUE NOT NULL,
    min_salary DECIMAL(10,2) CHECK(min_salary > 0),
    max_salary DECIMAL(10,2) CHECK(max_salary > 0),
    CHECK(max_salary > min_salary)
);


-- Real-world table with ALL constraint types combined
CREATE TABLE employees (
    employee_id SERIAL PRIMARY KEY,
    first_name VARCHAR(50) NOT NULL,
    last_name VARCHAR(50) NOT NULL,
    email VARCHAR(100) UNIQUE NOT NULL,
    phone VARCHAR(20) UNIQUE,
    hire_date DATE NOT NULL DEFAULT CURRENT_DATE,
    salary DECIMAL(10,2) NOT NULL CHECK(salary > 0),
    department_id INT NOT NULL REFERENCES departments(department_id)
        ON DELETE RESTRICT ON UPDATE CASCADE,
    job_id INT NOT NULL REFERENCES job_titles(job_id)
        ON DELETE RESTRICT ON UPDATE CASCADE,
    status VARCHAR(20) NOT NULL DEFAULT 'active'
        CHECK(status IN ('active', 'inactive', 'on_leave', 'terminated')),
    created_at TIMESTAMP NOT NULL DEFAULT NOW(),
    updated_at TIMESTAMP NOT NULL DEFAULT NOW(),
    CONSTRAINT chk_salary_range CHECK(salary BETWEEN 1000 AND 500000),
    CONSTRAINT uq_emp_name_dept UNIQUE(first_name, last_name, department_id)
);

-- CASCADE behavior explained:
-- ON DELETE RESTRICT: Cannot delete a department if employees exist in it
-- ON UPDATE CASCADE: If department_id changes, employee records update automatically

-- Insert referenced data first
INSERT INTO departments (dept_name, location)
VALUES ('Engineering', 'Building A'), ('Marketing', 'Building B');

INSERT INTO job_titles (title, min_salary, max_salary)
VALUES ('Developer', 3000, 15000), ('Manager', 5000, 25000);

-- Insert employee (uses DEFAULT for hire_date, status, created_at, updated_at)
INSERT INTO employees (first_name, last_name, email, salary, department_id, job_id)
VALUES ('John', 'Smith', 'john@company.com', 5000, 1, 1);
```

## Technical Vocabulary

| Term | Meaning | Khmer |
|------|---------|-------|
| CASCADE | Automatically propagate changes | ផ្សព្វផ្សាយការផ្លាស់ប្តូរ |
| RESTRICT | Prevent action if referenced | ការពារសកម្មភាព |
| SET NULL | Set to NULL when parent deleted | កំណត់ NULL នៅពេលលុប |
| Constraint combination | Multiple rules on one table | បន្សំច្បាប់ច្រើន |
| Data integrity | Accuracy and consistency of data | ភាពត្រឹមត្រូវទិន្នន័យ |
| Cascading action | Action that triggers related changes | សកម្មភាពបន្សាន |

## Speaking Practice
"In production databases, we combine multiple constraints to ensure complete data integrity. For example, our employees table uses NOT NULL to require essential fields, CHECK to validate salary ranges, UNIQUE to prevent duplicate emails, DEFAULT for automatic timestamps, and FOREIGN KEY with CASCADE rules to maintain relationships between tables."
