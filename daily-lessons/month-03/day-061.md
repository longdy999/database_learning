# Day 61: One-to-One Relationships

## Topic
One-to-One Relationships in Database Design

## English Explanation
A one-to-one (1:1) relationship means that one row in Table A relates to exactly one row in Table B, and vice versa. This is used when you want to split data into separate tables for organization, security, or performance reasons.

For example, each user has exactly one user profile. Each employee has exactly one parking spot assigned. You enforce this by placing a UNIQUE constraint on the foreign key column.

**When to use 1:1 relationships:**
- Separating sensitive data (passwords, salary) from general data
- Splitting a wide table into logical groups
- Optional data that not every row needs

## SQL Example

```sql
-- Create the parent table (users)
CREATE TABLE users (
    user_id INT PRIMARY KEY AUTO_INCREMENT,
    username VARCHAR(50) NOT NULL UNIQUE,
    email VARCHAR(100) NOT NULL UNIQUE,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- Create the child table with UNIQUE foreign key (1:1)
CREATE TABLE user_profiles (
    profile_id INT PRIMARY KEY AUTO_INCREMENT,
    user_id INT NOT NULL UNIQUE,
    full_name VARCHAR(100),
    date_of_birth DATE,
    phone_number VARCHAR(20),
    address TEXT,
    bio TEXT,
    FOREIGN KEY (user_id) REFERENCES users(user_id)
        ON DELETE CASCADE
        ON UPDATE CASCADE
);
```

**Key point:** The `UNIQUE` constraint on `user_id` in user_profiles ensures each user can only have ONE profile вҖ” this is what makes it 1:1 instead of 1:Many.

## Reading Practice

Read and understand this query that joins a 1:1 relationship:

```sql
SELECT u.username, u.email, p.full_name, p.phone_number
FROM users u
INNER JOIN user_profiles p ON u.user_id = p.user_id
WHERE u.username = 'sokha_chan';
```

**Question:** Why does this JOIN return exactly one row per user?
**Answer:** Because `user_id` in `user_profiles` has a UNIQUE constraint, so each user can match at most one profile row.
