# Day 78: CHECK Constraints - Data Validation

## Topic
Using CHECK Constraints to Validate Data at the Database Level

## English Explanation
A CHECK constraint enforces a rule on column values. If data violates the rule, the database rejects the INSERT or UPDATE. This ensures data quality without relying on application code alone.

**Common CHECK constraint patterns:**
- `CHECK(price > 0)` — price must be positive
- `CHECK(status IN ('active', 'inactive'))` — only allowed values
- `CHECK(age BETWEEN 1 AND 120)` — value must be in range
- `CHECK(end_date > start_date)` — compare two columns
- `CHECK(email LIKE '%@%.%')` — basic format validation

## Khmer Explanation
CHECK constraint បង្ខំច្បាប់លើតម្លៃជួរឈរ។ ប្រសិនបើទិន្នន័យបំពាន rule នោះ database បដិសេធ INSERT ឬ UPDATE។ នេះធានាគុណភាពទិន្នន័យដោយមិនពឹងផ្អែកលើ application code តែម្នាក់ឯង។

**CHECK constraint patterns ទូទៅ:**
- `CHECK(price > 0)` — តម្លៃត្រូវតែវិជ្ជមាន
- `CHECK(status IN ('active', 'inactive'))` — តម្លៃដែលអនុញ្ញាតតែប៉ុណ្ណោះ
- `CHECK(age BETWEEN 1 AND 120)` — តម្លៃត្រូវស្ថិតក្នុងជួរ
- `CHECK(end_date > start_date)` — ប្រៀបធៀបពីរជួរឈរ

## SQL Example
```sql
-- Basic CHECK constraints
CREATE TABLE products (
    product_id SERIAL PRIMARY KEY,
    product_name VARCHAR(100) NOT NULL,
    price DECIMAL(10,2) CHECK(price > 0),
    stock_qty INT CHECK(stock_qty >= 0),
    status VARCHAR(20) CHECK(status IN ('active', 'inactive', 'discontinued'))
);


-- CHECK with range
CREATE TABLE employees (
    employee_id SERIAL PRIMARY KEY,
    full_name VARCHAR(100) NOT NULL,
    age INT CHECK(age BETWEEN 18 AND 65),
    salary DECIMAL(10,2) CHECK(salary >= 0),
    hire_date DATE CHECK(hire_date >= '2000-01-01')
);

-- CHECK comparing two columns
CREATE TABLE events (
    event_id SERIAL PRIMARY KEY,
    event_name VARCHAR(100) NOT NULL,
    start_date DATE NOT NULL,
    end_date DATE NOT NULL,
    CHECK(end_date > start_date)
);

-- Named CHECK constraint (easier to manage)
CREATE TABLE accounts (
    account_id SERIAL PRIMARY KEY,
    balance DECIMAL(12,2),
    account_type VARCHAR(20),
    CONSTRAINT chk_balance_positive CHECK(balance >= 0),
    CONSTRAINT chk_valid_type CHECK(account_type IN ('savings', 'checking', 'business'))
);

-- Testing: These will FAIL
INSERT INTO products (product_name, price, stock_qty, status)
VALUES ('Bad Item', -5.00, 10, 'active');  -- FAILS: price must be > 0

INSERT INTO products (product_name, price, stock_qty, status)
VALUES ('Bad Item', 10.00, -1, 'active');  -- FAILS: stock_qty must be >= 0

INSERT INTO products (product_name, price, stock_qty, status)
VALUES ('Bad Item', 10.00, 5, 'unknown');  -- FAILS: invalid status

-- This will SUCCEED
INSERT INTO products (product_name, price, stock_qty, status)
VALUES ('Good Item', 25.99, 100, 'active');  -- OK
```

## Hands-on Practice

   - gpa must be between 0.0 and 4.0

## Speaking Practice
"CHECK constraints protect data quality at the database level. Even if the application has a bug, the database will reject invalid data. For example, we use CHECK(price > 0) to ensure no product can have a negative price. This is our last line of defense for data integrity."
