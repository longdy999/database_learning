# Day 70: Second Normal Form (2NF)

## Topic
Second Normal Form: eliminating partial dependencies in tables with composite keys.

## English Explanation
A table is in Second Normal Form (2NF) when: (1) it is already in 1NF, and (2) every non-key column depends on the ENTIRE primary key, not just part of it. Partial dependency only happens when you have a composite primary key (two or more columns). If a non-key column depends on only ONE part of the composite key, that is a partial dependency and violates 2NF. The fix is to move that column to a separate table.

## SQL Example

```sql
-- VIOLATES 2NF: partial dependency on composite key
-- PK is (student_id, course_id)
-- But student_name depends ONLY on student_id (partial dependency)
-- And course_name depends ONLY on course_id (partial dependency)

CREATE TABLE enrollments_bad (
    student_id INT,
    course_id INT,
    student_name VARCHAR(100),   -- depends only on student_id
    course_name VARCHAR(100),    -- depends only on course_id
    grade CHAR(2),               -- depends on BOTH (full dependency)
    PRIMARY KEY (student_id, course_id)
);

INSERT INTO enrollments_bad VALUES
(1, 101, 'Sokha', 'SQL Basics', 'A'),
(1, 102, 'Sokha', 'Web Dev', 'B'),
(2, 101, 'Dara', 'SQL Basics', 'A-');
-- Notice: 'Sokha' repeated, 'SQL Basics' repeated = redundancy!


-- CORRECT 2NF: remove partial dependencies to separate tables
CREATE TABLE students (
    student_id INT PRIMARY KEY,
    student_name VARCHAR(100)    -- depends fully on student_id
);

CREATE TABLE courses (
    course_id INT PRIMARY KEY,
    course_name VARCHAR(100)    -- depends fully on course_id
);

CREATE TABLE enrollments (
    student_id INT,
    course_id INT,
    grade CHAR(2),              -- depends on BOTH student_id AND course_id
    PRIMARY KEY (student_id, course_id),
    FOREIGN KEY (student_id) REFERENCES students(student_id),
    FOREIGN KEY (course_id) REFERENCES courses(course_id)
);

-- Now no redundancy: each name stored once
INSERT INTO students VALUES (1, 'Sokha'), (2, 'Dara');
INSERT INTO courses VALUES (101, 'SQL Basics'), (102, 'Web Dev');
INSERT INTO enrollments VALUES (1, 101, 'A'), (1, 102, 'B'), (2, 101, 'A-');
```

## Technical Vocabulary

| Term | Definition |
|------|-----------|
| Second Normal Form (2NF) | 1NF + no partial dependencies |
| Partial Dependency | Non-key column depends on only part of composite PK |
| Full Dependency | Non-key column depends on the entire primary key |
| Composite Key | Primary key made of two or more columns |
| Functional Dependency | Column B is determined by column A |
| Decomposition | Breaking a table into smaller, properly structured tables |
| Redundancy Elimination | Removing repeated data by proper table design |

## Speaking Practice
"This table has a composite primary key of student_id and course_id. The problem is that student_name depends only on student_id — it has nothing to do with which course. That is a partial dependency. To fix it, we move student_name to its own students table. Now each student name is stored exactly once, eliminating redundancy."
