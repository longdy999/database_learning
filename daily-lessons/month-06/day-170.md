## Section 2: Simple English Explanation
Code review is reading someone else's code and suggesting improvements. It is not
about embarrassing people — it is about making code better together.

## Section 4: SQL/Database Example
```sql
-- BEFORE review:
select * from users where status = 'active' and created_at > '2024-01-01'

-- AFTER review (improved):
SELECT u.id, u.name, u.email, u.created_at
FROM users u
WHERE u.status = 'active'
  AND u.created_at > '2024-01-01'
ORDER BY u.name ASC;
```

## Section 5: Hands-On Practice (10-15 minutes)
**Review these 3 queries and write feedback:**

### Query 1:
```sql
SELECT customers.name, orders.total, products.name
FROM customers, orders, products, order_items
WHERE customers.id = orders.customer_id
AND orders.id = order_items.order_id
AND order_items.product_id = products.id
AND orders.total > 100
```
**Feedback:**
- Uses implicit JOIN (comma FROM) — hard to read. Use explicit JOIN.
- Ambiguous column `name` from two tables — add aliases.
- No ORDER BY — results unpredictable.

### Query 2:
```sql
SELECT department, COUNT(*) FROM employees
GROUP BY department HAVING COUNT(*) > 5
```
**Feedback:**
- Good: Clean GROUP BY with HAVING.
- Add alias: `COUNT(*) AS employee_count`
- Add ORDER BY for useful output ordering.


### Query 3:
```sql
UPDATE products SET price = price * 1.1;
```
**Feedback:**
- Critical: No WHERE clause — updates ALL products!
- Run as SELECT first to verify affected rows.
- Wrap in transaction for safety.

## Section 6: Technical Vocabulary
| Term | Meaning |
|------|---------|
| Code review | Examining others' code for quality |
| Pull request | A request to merge changes, triggers review |
| Constructive feedback | Suggestions that help improve, not criticize |
| Implicit JOIN | Old-style comma FROM (avoid) |
| Migration script | SQL that changes schema in controlled way |

## Section 7: Speaking Practice
**Scenario:** You found a problem in a colleague's SQL.
Practice saying: "This query uses SELECT * which pulls more data than needed.
Could we list specific columns? Also, an index on status would help performance."

## Section 8: Reading/Listening Practice
- Go to any open-source GitHub project and read a pull request review
- Notice the tone and patterns: "Consider...", "Nit:", "Blocking:"
