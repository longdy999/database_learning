# Day 57: Query Debugging - Finding and Fixing Errors

## Topic
Query debugging: identifying common SQL errors, understanding error messages, and systematically fixing broken queries.

## English Explanation
Writing SQL is one skill. Debugging broken SQL is another essential skill. In real work, you will encounter errors daily — from your own queries and from code written by others. Common errors include missing JOIN conditions, incorrect GROUP BY clauses, using aggregates in WHERE instead of HAVING, and ambiguous column references. Today you learn to read error messages carefully, identify the root cause, and fix queries systematically instead of guessing.

## SQL Example

### Error 1: Missing ON Clause in JOIN
```sql
-- BROKEN: Missing join condition
SELECT customers.name, orders.total_amount
FROM customers
JOIN orders;
-- Error: Every JOIN needs an ON clause

-- FIXED:
SELECT customers.name, orders.total_amount
FROM customers
JOIN orders ON customers.customer_id = orders.customer_id;
```

### Error 2: Wrong GROUP BY (missing non-aggregated column)
```sql
-- BROKEN: customer_name is not in GROUP BY
SELECT customer_name, category, SUM(amount) AS total
FROM orders
GROUP BY category;
-- Error: customer_name must be in GROUP BY or inside aggregate


-- FIXED: Include all non-aggregated columns in GROUP BY
SELECT customer_name, category, SUM(amount) AS total
FROM orders
GROUP BY customer_name, category;
```

### Error 3: Aggregate Function in WHERE (should use HAVING)
```sql
-- BROKEN: Cannot use SUM() in WHERE
SELECT category, SUM(amount) AS total
FROM orders
WHERE SUM(amount) > 1000
GROUP BY category;
-- Error: Aggregate functions not allowed in WHERE

-- FIXED: Use HAVING for aggregate conditions
SELECT category, SUM(amount) AS total
FROM orders
GROUP BY category
HAVING SUM(amount) > 1000;
```

### Error 4: Ambiguous Column Reference
```sql
-- BROKEN: Both tables have 'id' column
SELECT id, name, order_date
FROM customers
JOIN orders ON customers.customer_id = orders.customer_id;
-- Error: Column 'id' is ambiguous

-- FIXED: Use table aliases to clarify
SELECT c.customer_id, c.name, o.order_date
FROM customers c
JOIN orders o ON c.customer_id = o.customer_id;
```

### Error 5: Using Alias in WHERE (alias not yet available)
```sql
-- BROKEN: Cannot use column alias in WHERE
SELECT name, price * quantity AS total_cost
FROM order_items
WHERE total_cost > 100;
-- Error: Unknown column 'total_cost'

-- FIXED: Repeat the expression or use subquery
SELECT name, price * quantity AS total_cost
FROM order_items
WHERE price * quantity > 100;
```

## Speaking Practice
"When debugging SQL queries, I follow a systematic approach. First, I read the error message carefully — it usually tells me the exact line and type of problem. Common issues include missing JOIN conditions, aggregate functions used in WHERE instead of HAVING, and ambiguous column names. I fix these by adding proper ON clauses, moving conditions to HAVING, and using table aliases."


## Reading Practice
Read the following error messages and identify the problem:

1. `ERROR: column "name" must appear in the GROUP BY clause or be used in an aggregate function`
   → **Fix:** Add `name` to GROUP BY or wrap it in an aggregate

2. `ERROR: column reference "id" is ambiguous`
   → **Fix:** Prefix with table name or alias (e.g., `customers.id`)

3. `ERROR: aggregate functions are not allowed in WHERE`
   → **Fix:** Move the condition to HAVING clause

4. `ERROR: missing FROM-clause entry for table "c"`
   → **Fix:** Check that table alias "c" is defined in FROM/JOIN

5. `ERROR: syntax error at or near "JOIN"`
   → **Fix:** Check for missing comma, keyword, or incomplete previous clause
