# Day 38: Why JOINs Matter - Connecting Tables with Relationships

## Topic
Understanding why databases use multiple tables and how foreign keys connect them together.

## English Explanation
In a real database, information is split across multiple tables. A customer's name is in the customers table, but their orders are in a separate orders table. Why? Because storing the customer's full name and address in every single order row would waste space and cause update problems. Instead, we connect tables using keys. The orders table has a customer_id column that points back to the customers table. JOIN is the SQL command that reconnects these tables when we need combined information.

## SQL Example

```sql
-- Two related tables:

-- Table 1: customers
-- | id | name        | email              | city      |
-- |----|-------------|--------------------|-----------|
-- | 1  | Sokha       | sokha@mail.com     | Phnom Penh|
-- | 2  | Dara        | dara@mail.com      | Siem Reap |
-- | 3  | Bopha       | bopha@mail.com     | Battambang|

-- Table 2: orders
-- | id | customer_id | product      | amount | order_date |
-- |----|-------------|--------------|--------|------------|
-- | 1  | 1           | Laptop       | 1200   | 2024-01-15 |
-- | 2  | 1           | Mouse        | 25     | 2024-01-16 |
-- | 3  | 2           | Keyboard     | 75     | 2024-01-17 |
-- | 4  | 3           | Monitor      | 350    | 2024-01-18 |
```


```sql
-- The RELATIONSHIP: orders.customer_id вҶ’ customers.id
-- This is called a FOREIGN KEY

-- Without JOIN, you only see customer_id (a number):
SELECT * FROM orders;
-- You see customer_id = 1, but who is customer 1?

-- With JOIN, you get the full picture:
SELECT customers.name, orders.product, orders.amount
FROM orders
JOIN customers ON orders.customer_id = customers.id;
-- Result: Sokha | Laptop | 1200

-- Visual explanation of how JOIN works:
-- Step 1: Take a row from orders (customer_id = 1)
-- Step 2: Find the matching row in customers (id = 1)
-- Step 3: Combine both rows into one result row
-- Step 4: Repeat for every row in orders
```

## Speaking Practice

Explain to a client:
"Your database stores customers in one table and orders in another. They are
connected by a customer ID. This design means if a customer changes their
address, we only update it in one place вҖ” not in hundreds of order records.
When we need a report showing customer names with their orders, we use JOIN
to connect the tables temporarily for that query."

## Reading Practice

Read this CREATE TABLE statement and identify the relationship:

```sql
CREATE TABLE customers (
    id INT PRIMARY KEY,
    name VARCHAR(100),
    email VARCHAR(100)
);

CREATE TABLE orders (
    id INT PRIMARY KEY,
    customer_id INT,
    product VARCHAR(100),
    amount DECIMAL(10,2),
    FOREIGN KEY (customer_id) REFERENCES customers(id)
);
```

Questions:
- Which column is the foreign key?
- Which table does it reference?
- Can an order exist without a valid customer_id? Why or why not?
