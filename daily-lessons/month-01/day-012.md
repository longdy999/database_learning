# Day 12: NOT NULL and DEFAULT Values

## Topic
Understanding NOT NULL to prevent empty values, and DEFAULT to provide automatic values when none are specified.

## English Explanation
**NOT NULL** and **DEFAULT** are two constraints that work together to control what happens when data is inserted:
- **NOT NULL** means a column must always receive a value вҖ” you cannot skip it or leave it empty
- **DEFAULT** provides an automatic fallback value if you do not specify one during INSERT

When you combine NOT NULL with DEFAULT, the column always has a value: either the one you provide, or the default. This keeps your data clean and complete without forcing users to fill in every field manually.

**Key difference:**
- NOT NULL without DEFAULT = you MUST provide a value every time
- DEFAULT without NOT NULL = automatic value is used, but NULL is still allowed
- NOT NULL + DEFAULT = always has a value (yours or the default)

## SQL Example
```sql
-- Table showing NOT NULL and DEFAULT working together
CREATE TABLE orders (
    order_id INTEGER PRIMARY KEY AUTOINCREMENT,
    customer_name TEXT NOT NULL,
    product TEXT NOT NULL,
    quantity INTEGER NOT NULL DEFAULT 1,
    status TEXT NOT NULL DEFAULT 'pending',
    created_date TEXT NOT NULL DEFAULT (date('now')),
    notes TEXT DEFAULT 'No special instructions'
);


-- INSERT with all values provided
INSERT INTO orders (customer_name, product, quantity, status, created_date, notes)
VALUES ('Sokha', 'Laptop', 2, 'confirmed', '2024-06-15', 'Gift wrap please');

-- INSERT relying on DEFAULT values
INSERT INTO orders (customer_name, product)
VALUES ('Dara', 'Keyboard');
-- Result: quantity=1, status='pending', created_date=today, notes='No special instructions'

-- INSERT with some defaults overridden
INSERT INTO orders (customer_name, product, quantity, status)
VALUES ('Vanna', 'Mouse', 3, 'shipped');
-- Result: created_date=today, notes='No special instructions'

-- This FAILS: customer_name is NOT NULL with no DEFAULT
INSERT INTO orders (product)
VALUES ('Monitor');
-- ERROR: NOT NULL constraint failed: orders.customer_name

SELECT * FROM orders;
```

## Hands-on Practice (10-15 minutes)
Create these two tables and test DEFAULT behavior:
```sql
-- Table 1: User accounts with status defaults
CREATE TABLE user_accounts (
    user_id INTEGER PRIMARY KEY AUTOINCREMENT,
    username TEXT UNIQUE NOT NULL,
    email TEXT UNIQUE NOT NULL,
    role TEXT NOT NULL DEFAULT 'member',
    is_active INTEGER NOT NULL DEFAULT 1,
    created_at TEXT NOT NULL DEFAULT (datetime('now')),
    last_login TEXT DEFAULT NULL
);

-- Table 2: Products with inventory defaults
CREATE TABLE products (
    product_id INTEGER PRIMARY KEY AUTOINCREMENT,
    name TEXT NOT NULL,
    category TEXT NOT NULL DEFAULT 'Uncategorized',
    price REAL NOT NULL,
    stock_quantity INTEGER NOT NULL DEFAULT 0,
    is_available INTEGER NOT NULL DEFAULT 1,
    description TEXT DEFAULT 'No description provided'
);


-- Test: Insert with minimum required fields
INSERT INTO user_accounts (username, email)
VALUES ('sokha_dev', 'sokha@email.com');

INSERT INTO products (name, price)
VALUES ('Wireless Mouse', 25.99);

-- Check what DEFAULT filled in
SELECT * FROM user_accounts;
SELECT * FROM products;
```

## Speaking Practice
"In this table design, I used NOT NULL on customer_name because every order must belong to someone. I added DEFAULT 'pending' on status so new orders automatically start as pending without the user needing to specify it. The created_date defaults to today's date. This design makes the system easier to use while keeping data complete."
