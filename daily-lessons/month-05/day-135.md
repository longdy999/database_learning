# Day 135: Database Security Fundamentals

## Topic
Database Security Fundamentals — The CIA triad, authentication vs authorization, and common database threats.

## English Explanation
Database security protects your data from unauthorized access, modification, or destruction. The foundation is the CIA triad: Confidentiality (only authorized people see data), Integrity (data cannot be changed without permission), and Availability (data is accessible when needed). Authentication verifies identity ("who are you?") while Authorization controls permissions ("what can you do?"). Common threats include SQL injection (malicious code in queries), unauthorized access (weak passwords or missing permissions), and data exposure (sensitive data visible to wrong people).

## SQL Example

```sql
-- AUTHENTICATION: Creating users with passwords
CREATE USER app_reader WITH PASSWORD 'secure_password_123';
CREATE USER app_writer WITH PASSWORD 'another_secure_pwd';
CREATE USER admin_user WITH PASSWORD 'admin_strong_pwd!';

-- AUTHORIZATION: Granting specific permissions
-- Read-only user: can only SELECT
GRANT SELECT ON ALL TABLES IN SCHEMA public TO app_reader;

-- Writer user: can SELECT, INSERT, UPDATE
GRANT SELECT, INSERT, UPDATE ON ALL TABLES IN SCHEMA public TO app_writer;

-- Admin: full access
GRANT ALL PRIVILEGES ON ALL TABLES IN SCHEMA public TO admin_user;

-- Revoking permissions
REVOKE DELETE ON customers FROM app_writer;

-- SQL INJECTION EXAMPLE (what NOT to do):
-- BAD: String concatenation in application code
-- query = "SELECT * FROM users WHERE name = '" + user_input + "'"
-- If user_input = "'; DROP TABLE users; --"
-- Result: SELECT * FROM users WHERE name = ''; DROP TABLE users; --'
```


```sql
-- SAFE: Using parameterized queries (application level)
-- query = "SELECT * FROM users WHERE name = $1"
-- parameters = [user_input]

-- Viewing current user permissions
SELECT grantee, privilege_type, table_name
FROM information_schema.table_privileges
WHERE table_schema = 'public'
ORDER BY grantee, table_name;

-- Creating a role for a team
CREATE ROLE reporting_team;
GRANT SELECT ON orders, customers, products TO reporting_team;
GRANT reporting_team TO app_reader;
```

## Hands-on Practice
Identify security vulnerabilities in a sample system:

```sql
-- Sample system with security problems:
-- Problem 1: All users have full admin access
-- Problem 2: Passwords stored in plain text
-- Problem 3: No input validation (SQL injection risk)
-- Problem 4: Sensitive data (SSN, credit cards) not encrypted
-- Problem 5: No audit log of who accessed what

-- Fix 1: Create role-based access
CREATE ROLE readonly_role;
CREATE ROLE readwrite_role;
CREATE ROLE admin_role;

GRANT SELECT ON ALL TABLES IN SCHEMA public TO readonly_role;
GRANT SELECT, INSERT, UPDATE ON ALL TABLES IN SCHEMA public TO readwrite_role;
GRANT ALL PRIVILEGES ON ALL TABLES IN SCHEMA public TO admin_role;

-- Fix 2: Assign users to appropriate roles
GRANT readonly_role TO report_user;
GRANT readwrite_role TO app_user;
GRANT admin_role TO dba_user;

-- Fix 3: Restrict access to sensitive columns
REVOKE ALL ON sensitive_data FROM PUBLIC;
CREATE VIEW safe_customer_view AS
SELECT id, name, email, city FROM customers;
GRANT SELECT ON safe_customer_view TO readonly_role;
```

## Speaking Practice

**Explain to a client:**
"Database security has three pillars: keeping data confidential so only authorized people can see it, maintaining data integrity so it cannot be tampered with, and ensuring availability so the system works when you need it. I recommend implementing role-based access control — each user gets only the permissions they need for their job. We also need to protect against SQL injection attacks by using parameterized queries in our application code."
