# Day 137: SQL Injection Prevention

## Topic
Understanding SQL injection attacks and how to prevent them using parameterized queries and prepared statements.

## English Explanation
SQL injection is one of the most dangerous and common database attacks. It happens when an attacker inserts malicious SQL code into input fields that get executed by the database. For example, if a login form directly concatenates user input into a query, an attacker can type special characters to bypass authentication or delete data. Parameterized queries (also called prepared statements) solve this by separating SQL logic from user data — the database treats input as data only, never as executable code.

## SQL Example
```sql
-- VULNERABLE query (NEVER do this!)
-- User input: ' OR '1'='1' --
SELECT * FROM users
WHERE username = '' OR '1'='1' --' AND password = 'anything';
-- This returns ALL users because '1'='1' is always true!

-- ANOTHER dangerous input: '; DROP TABLE users; --
SELECT * FROM users
WHERE username = ''; DROP TABLE users; --';
-- This could delete your entire users table!

-- SAFE: Parameterized query (PostgreSQL example)
PREPARE login_query (text, text) AS
  SELECT * FROM users WHERE username = $1 AND password = $2;
EXECUTE login_query('john_doe', 'hashed_password_here');

-- SAFE: Using prepared statements in application code (pseudocode)
-- stmt = db.prepare("SELECT * FROM users WHERE username = ? AND password = ?")
-- stmt.execute(user_input_name, user_input_pass)
```

## Technical Vocabulary

| Term | Definition |
|------|-----------|
| SQL Injection | Attack where malicious SQL is inserted via user input |
| Parameterized Query | Query where user data is passed as parameters, not concatenated |
| Prepared Statement | Pre-compiled query template that accepts parameters safely |
| Input Validation | Checking user input for expected format before processing |
| Sanitization | Cleaning user input to remove dangerous characters |
| Escaping | Converting special characters so they are treated as text |
| Attack Vector | The method an attacker uses to exploit a vulnerability |
| Payload | The malicious code an attacker sends |
| Whitelist | Allowing only known-good input values |
| ORM | Object-Relational Mapping, often provides built-in injection protection |

## Speaking Practice
"We never concatenate user input directly into SQL queries. Instead, we use parameterized queries where the database engine treats all input strictly as data. This completely prevents SQL injection attacks, which remain one of the top security vulnerabilities in web applications."
