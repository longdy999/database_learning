# Day 138: Data Encryption and Sensitive Data Handling

## Topic
Encryption at rest vs in transit, password hashing, column-level encryption, and PII handling best practices.

## English Explanation
Sensitive data needs protection at every stage. Encryption at rest means data stored on disk is encrypted — if someone steals the hard drive, they cannot read the data. Encryption in transit means data traveling over the network is encrypted using SSL/TLS — if someone intercepts the connection, they see only scrambled data. Passwords must NEVER be stored as plain text; instead, use one-way hashing algorithms (like bcrypt) so even database administrators cannot see actual passwords. PII (Personally Identifiable Information) like names, emails, and ID numbers needs special handling to comply with privacy laws.

## SQL Example
```sql
-- WRONG: Storing plain text password (NEVER do this!)
INSERT INTO users (username, password)
VALUES ('john', 'mypassword123');

-- CORRECT: Store hashed password (application handles hashing)
-- In PostgreSQL with pgcrypto extension:
CREATE EXTENSION IF NOT EXISTS pgcrypto;

INSERT INTO users (username, password_hash)
VALUES ('john', crypt('mypassword123', gen_salt('bf')));

-- Verify password at login
SELECT username FROM users
WHERE username = 'john'
  AND password_hash = crypt('mypassword123', password_hash);

-- Column-level encryption for sensitive data
INSERT INTO customers (name, email_encrypted)
VALUES ('Jane Doe', pgp_sym_encrypt('jane@email.com', 'encryption_key'));

-- Decrypt when needed
SELECT name, pgp_sym_decrypt(email_encrypted::bytea, 'encryption_key') AS email
FROM customers WHERE name = 'Jane Doe';
```

## Technical Vocabulary

| Term | Definition |
|------|-----------|
| Encryption at Rest | Data encrypted while stored on disk |
| Encryption in Transit | Data encrypted while traveling over a network |
| Hashing | One-way transformation; cannot be reversed to original |
| bcrypt | A strong password hashing algorithm with built-in salting |
| Salt | Random data added before hashing to prevent rainbow table attacks |
| PII | Personally Identifiable Information (name, email, SSN, etc.) |
| SSL/TLS | Protocols that encrypt network communications |
| pgcrypto | PostgreSQL extension for cryptographic functions |
| Plain Text | Unencrypted, readable data (dangerous for passwords) |
| Data Masking | Showing partial data (e.g., ****1234) for security |

## Speaking Practice
"We never store passwords in plain text. All passwords are hashed using bcrypt before storage, so even if our database is compromised, attackers cannot recover the original passwords. For sensitive PII like email addresses, we use column-level encryption and restrict access to authorized personnel only."
