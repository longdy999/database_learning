# Day 122: PostgreSQL Data Types and Differences from SQLite

## Topic
PostgreSQL Data Types and Differences from SQLite - Understanding PostgreSQL's strict type system, unique data types, and type casting syntax.

## English Explanation
PostgreSQL enforces strict data types unlike SQLite which uses dynamic typing. In SQLite, you can store text in an INTEGER column without error, but PostgreSQL will reject this. PostgreSQL offers rich data types: VARCHAR(n) for limited-length text, TEXT for unlimited text, SERIAL for auto-increment, BOOLEAN for true/false values, DATE and TIMESTAMP for time data, NUMERIC for precise decimals, and JSONB for structured JSON data. The `::type` syntax is PostgreSQL's shorthand for type casting (converting one type to another). For example, `SELECT '42'::INTEGER` converts text to a number. Understanding these types is essential for designing reliable database schemas.

## SQL Example
```sql
-- PostgreSQL Data Types Demonstration

-- VARCHAR vs TEXT
CREATE TABLE type_demo (
    short_name VARCHAR(50),      -- max 50 characters
    description TEXT,             -- unlimited length
    age INTEGER,                  -- whole numbers only
    price NUMERIC(8, 2),         -- 8 digits total, 2 decimal
    is_available BOOLEAN,         -- TRUE or FALSE only
    created_at TIMESTAMP DEFAULT NOW(),
    birth_date DATE,
    tags TEXT[],                  -- array of text
    settings JSONB                -- binary JSON
);

-- SERIAL (auto-increment) - replaces SQLite's AUTOINCREMENT
CREATE TABLE products (
    id SERIAL PRIMARY KEY,       -- auto-generates 1, 2, 3...
    name VARCHAR(100) NOT NULL
);

-- Type casting with :: syntax
SELECT '2024-01-15'::DATE;                    -- text to date
SELECT 42::TEXT;                               -- integer to text
SELECT '3.14'::NUMERIC;                       -- text to numeric
SELECT '{"key": "value"}'::JSONB;             -- text to JSONB
SELECT EXTRACT(YEAR FROM NOW()::TIMESTAMP);   -- extract year
```

## Speaking Practice
"PostgreSQL uses strict data types, so every column must receive data that matches its declared type. For example, if a column is INTEGER, you cannot store text in it. This prevents data corruption. We also use the double-colon syntax to cast between types when needed, like converting a text date into a proper DATE type for comparison."
